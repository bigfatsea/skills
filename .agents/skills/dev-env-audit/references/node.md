# Node.js — Authoritative Manager: fnm + pnpm

## 1. Baseline

- Recommended tools: **fnm** (version management, 10–40x faster than nvm) + **pnpm** (package management, installed via the official standalone installer).
- Alternative for multi-language full-stack unified scenarios: Users already managing other languages with ASDF, who value “one tool, one mental model” over ultimate speed, can use the `asdf-nodejs` plugin instead of fnm. This is a trade-off of “convenience vs. speed”, not an unconditional upgrade—ASDF relies on shim forwarding, while fnm switches via environment variables directly; for pure Node scenarios, fnm remains faster with lower startup overhead. Do not install both simultaneously.
- Minimum qualifying feature set:
  - `node` resolves to a path managed by fnm (or asdf); `which -a node` shows no brew/system copies
  - `command -v pnpm` resolves to the standalone installation path (does not contain `fnm_multishells`, see §5)
  - Initialization code for nvm / volta / n is not active in shell configuration

## 2. Deep Dive Probe (read-only)

```bash
which -a node npm pnpm

# old managers
ls -d ~/.nvm 2>/dev/null
command -v volta; ls -d ~/.volta 2>/dev/null
command -v n; ls -d /usr/local/n 2>/dev/null

# init residue (first check source chain of scan.sh; centralized dotfiles frameworks may place init code
# into sourced files; grep failing on literal rc files doesn't mean absent, include files on source chain
# as targets for the grep below)
grep -n 'NVM_DIR\|nvm.sh' ~/.zshrc ~/.zprofile ~/.zshenv ~/.bash_profile ~/.bashrc 2>/dev/null
grep -n 'VOLTA_HOME' ~/.zshrc ~/.zprofile ~/.zshenv 2>/dev/null

# brew node (conflicts with fnm; must truly uninstall, not just disable)
brew list --formula 2>/dev/null | grep -E '^node(@[0-9]+)?$'

# fnm status
fnm --version; fnm list 2>/dev/null; fnm current 2>/dev/null

# pnpm: path is key—is it a shim generated by corepack in the fnm version directory?
command -v pnpm
pnpm config get store-dir 2>/dev/null   # store location is only trusted from this command; environment variables can be misleading

# global package list (record before migration, reinstall one by one afterwards)
npm ls -g --depth=0 2>/dev/null

# NPM_CONFIG_PREFIX has two layers: beyond the shell variable, there may be a separate persistent entry in ~/.npmrc
# (from a previous npm config set prefix ...); checking only env will miss this layer, see §5
echo "env: ${NPM_CONFIG_PREFIX:-<unset>}"
grep -n '^prefix' ~/.npmrc 2>/dev/null
```

## 3. Judgment Rules

| Finding | Verdict | Reason |
|---|---|---|
| node managed by fnm, pnpm installed standalone and unique | OK | Meets requirements |
| brew node coexists with fnm | FAIL | Two node binaries conflict on PATH; must uninstall brew node |
| nvm/volta/n init code still active | FAIL | Conflicts with fnm on PATH/hook |
| ~/.nvm etc. directories remain but init code removed | WARN | Harmless residue; can be deleted after confirmation |
| `command -v pnpm` contains `fnm_multishells` or fnm version directory | WARN | corepack shim has hijacked standalone pnpm (see §5); adjust PATH order |
| Only brew node, no any manager | WARN | Usable but not recommended; low risk for single-version users, inevitable trouble for multi-version projects |
| Mixing yarn/npm/pnpm global packages | WARN | Recommend consolidating global packages under pnpm |
| NPM_CONFIG_PREFIX is set and a version manager (fnm/nvm etc.) is in use | WARN | Global packages break version isolation; native modules break when switching Node versions (see §5); recommend removing this variable to let fnm restore default per-version isolation—also check `~/.npmrc` for an independent persistent `prefix=` (both must be removed to take effect, see §5). “Migrate global tools to pnpm” is not a solution for this issue, as pnpm’s global directory is also a fixed path and does not provide per-version isolation |
| grep on top-level rc files misses, but source chain files contain NVM_DIR/VOLTA_HOME | FAIL | Logic hidden in centrally managed files; it’s there, not absent |

## 4. Migration Plan (five steps)

1. **Inspect current state** — Full §2 probe; **first record global packages with `npm ls -g --depth=0`**.
2. **Install fnm + pnpm**:
   ```bash
   curl -fsSL https://fnm.vercel.app/install | bash     # or brew install fnm
   curl -fsSL https://get.pnpm.io/install.sh | sh -     # pnpm via the official standalone installer; do not rely on corepack
   ```
   The pnpm install script by default appends configuration to `~/.zshrc`—if the user uses centralized config management, remind them to check and move the content to their own configuration file.
   ```bash
   fnm install --lts
   fnm default <version>
   ```
3. **Handle old managers**:
   - brew node: **Must truly uninstall** `brew uninstall node` (leaving it will definitely hijack PATH).
   - nvm/volta/n: Delete or comment out the corresponding initialization code in shell config; no need to uninstall the binaries themselves.
   - Global npm packages: Reinstall one by one using the list from step 1 with `pnpm install -g <pkg>`; after verifying the pnpm version works, use `npm uninstall -g <pkg>` to clean up the old copies (destructive; confirm each).
4. **[Optional] External storage**:
   ```bash
   export FNM_DIR="/Volumes/<disk>/dev-cache/fnm"    # Location for Node version bodies; set before installing the first version
   export NPM_CONFIG_CACHE="/Volumes/<disk>/dev-cache/npm-cache"
   pnpm config set store-dir "/Volumes/<disk>/dev-cache/pnpm-store"   # store must be changed with this command; env vars or moving the directory won't work
   ```
   **Do not set `NPM_CONFIG_PREFIX`**: it pins npm global packages into a single directory, breaking fnm’s per‑Node‑version isolation (see §5). If already set, remove it—**but check two places**: besides `export NPM_CONFIG_PREFIX=...` in shell, there may be a separate persistent `prefix=...` line in `~/.npmrc` (from a previous `npm config set prefix ...`); if you only delete the env var but not that line, npm will still read the old value. Run `grep -n '^prefix' ~/.npmrc` first. “Move global tools to pnpm” is not a fix for this, see §5.
5. **Verify**:
   ```bash
   node -v; npm -v; pnpm -v
   command -v node pnpm      # node should be in fnm path; pnpm should not contain fnm_multishells
   pnpm config get store-dir
   ```

## 5. Known Pitfalls

- **corepack pnpm shim**: If you ever run `corepack pnpm ...` under a Node managed by fnm, corepack generates a pnpm shim in **that Node version’s own bin directory**. This is a different binary from the standalone pnpm installation; which one takes effect depends on PATH order. If `command -v pnpm` output contains `fnm_multishells`, you’ve been hit—move the standalone pnpm installation directory (e.g., `~/Library/pnpm`) before the fnm directory in PATH.
- **pnpm store location three no’s**: it doesn’t honor environment variables, doesn’t honor direct `mv` of the directory, only honors `pnpm config set store-dir`.
- **brew node must be uninstalled, not just disabled**: it installs node/npm into `/opt/homebrew/bin`, which is always on PATH; there is no init code to remove, so the only solution is to uninstall.
- **pnpm install script adds content to ~/.zshrc**: After installation, check whether a config block has been appended.
- **NPM_CONFIG_PREFIX conflicts with version managers**: With a fixed global prefix, `npm -g` installed native modules are bound to the ABI of the Node version at install time; fnm’s version switching breaks them; nvm simply detects it and refuses to work. The real solution is to remove this variable and let fnm restore its default “each Node version has its own global directory”—**not “switch to pnpm”**: pnpm’s global directory (`pnpm root -g`, which does not change with fnm’s current Node version in practice) is the same structure as a fixed npm prefix (a single, Node‑version‑agnostic fixed directory), and offers no ABI immunity for packages with native bindings; it merely moves the risk from npm’s directory to pnpm’s. To decide whether a global package needs concern, check whether it contains `.node` files/`binding.gyp` (pure JS packages or self‑contained executables not loaded via `require()` are unaffected).
- **NPM_CONFIG_PREFIX has two layers, not one**: beyond `export NPM_CONFIG_PREFIX=...` (shell variable), `npm config set prefix <path>` persists the same value into `~/.npmrc` as a `prefix=` line—this is a second source independent of the shell environment. When removing, if you only delete the shell variable without checking `~/.npmrc`, npm will still read the old value from `~/.npmrc`. `npm config get prefix` shows only the effective value, not which layer it comes from; use `grep -n '^prefix' ~/.npmrc` or `npm config list -l` to locate it. Both checks must be included in judgment and migration steps (already incorporated into §2/§3/§4).
